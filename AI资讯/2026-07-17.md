## 🧠 大模型与 LLM 技术解读

1. 🔥 **GPT-5.6：从单模型能力到可编排 Agent 计算**
   - 背景：前沿模型已经从聊天和单次代码生成，进入长程知识工作、terminal agent、browser agent、cyber/science workflow 竞争。
   - 问题：瓶颈在于复杂任务的 token 成本、工具调用轮次、长程任务稳定性和多代理协调时延。
   - 方法：OpenAI 发布 Sol/Terra/Luna 三档模型，加入 Programmatic Tool Calling，让模型可用轻量程序过滤工具中间结果；`ultra` 模式默认协调 4 个 agents 并行推进高难任务。
   - 效果：OpenAI 声称 Sol 在 Agents' Last Exam 达到 53.6，coding agent index 达 80，并在 BrowseComp、OSWorld、ExploitBench 等评测上提升；这些主要来自官方与合作伙伴评测，仍需独立复现。来源：[OpenAI GPT-5.6](https://openai.com/index/gpt-5-6/)

2. 🔥 **Kimi K3：开放 2.8T MoE 模型把“开放权重”推向工程可用前沿**
   - 背景：GPT-5.6、Claude Fable、Gemini 3.5 等商业模型提升后，开放模型需要证明在 long-horizon coding、科学计算和本地/私有部署上的价值。
   - 问题：超大 MoE 的主要瓶颈是 attention 长上下文效率、expert routing 稳定性、训练吞吐和推理成本。
   - 方法：Kimi K3 使用 Kimi Delta Attention、Attention Residuals、Stable LatentMoE，2.8T 参数中激活 16/896 experts，并引入 quantization-aware training 与大规模 expert-parallel 训练/推理方案。
   - 效果：官方称整体 scaling efficiency 较 Kimi K2 提升约 2.5x，并给出 Terminal-Bench、DeepSWE、GPU kernel/compiler 等案例；完整技术报告和权重尚未发布，暂无独立评测。来源：[Kimi K3 技术博客](https://www.kimi.com/blog/kimi-k3)

3. ⭐ **Gemini Notebook：RAG notebook 正在变成带执行环境的研究 Agent**
   - 背景：NotebookLM 已经从资料问答扩展到 audio/video/slides 等多模态总结，Google 现在将其纳入 Gemini 品牌。
   - 问题：传统 notebook/RAG 工具只能“读资料并回答”，难以执行代码、验证数据、生成可复用 artifacts。
   - 方法：Google 将 NotebookLM 更名为 Gemini Notebook，并为高阶用户提供 secure cloud computer，让 notebook 可以在隔离环境中写和运行代码，同时与 Gemini/Search 生态更紧密连接。
   - 效果：Google 官方强调产品仍独立、现有 notebooks 保留；执行环境能力目前按订阅层级推出，真实性能和安全边界还需要用户侧验证。来源：[Google Blog](https://blog.google/innovation-and-ai/products/gemini-notebook/notebooklm-gemini-notebook/)

4. ⭐ **DeepMind bioresilience：前沿模型能力评估进入生物安全闭环**
   - 背景：AGI/前沿模型治理讨论正从抽象安全原则转向 biosecurity、cybersecurity 等高风险能力评测。
   - 问题：同一类模型既可能辅助药物/疫苗研发，也可能降低生物风险相关知识门槛；难点是如何让能力服务防御而非滥用。
   - 方法：DeepMind 与 Isomorphic Labs 推出 bioresilience program，方向包括病原监测、疫苗与疗法开发、疫情响应，并配合安全缓解措施。
   - 效果：这是项目发布和治理表态，暂无独立量化评测；但它说明 frontier lab 正把 bio domain 作为模型部署门槛的一部分。来源：[Axios](https://www.axios.com/2026/07/16/google-deepmind-biosecurity-safety)

5. 📌 **LLM 评测可信度成为新论文热点：judge、forecasting、agent benchmark 都在被反思**
   - 背景：模型发布越来越依赖 benchmark 和 LLM-as-judge，但 agentic workflows 的真实可迁移性很难用单一分数说明。
   - 问题：LLM-as-judge 数据集可能存在 test oracle 失真，forecasting/agent harness 也可能被 benchmark 设计限制。
   - 方法：近期 arXiv 论文覆盖 prediction market replay、synthetic LLM-as-judge validation、agent optimizer continual-learning、self-evolving agent harness 等方向。
   - 效果：这些论文主要提供评测框架与问题定义，尚不是产业级结论；但它们提示使用 GPT-5.6/Kimi K3 等新模型时应关注评测设计、闭环优化和真实任务复现。来源：[Hindcast](https://arxiv.org/abs/2607.14051)、[The Test Oracle Problem](https://arxiv.org/abs/2607.13707)、[Do Agent Optimizers Compound?](https://arxiv.org/abs/2607.14004)
