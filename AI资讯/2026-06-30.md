## 🧠 大模型与 LLM 技术解读

### 1. 🔥 RiVER：没有标准答案也能做可验证强化学习
- **背景**：RL with verifiable rewards（RLVR）通常依赖数学或代码任务的唯一答案。
- **问题**：组合优化等任务只有连续执行分数，没有 ground truth；原始分数还会产生跨实例 scale dominance 和重复采样的 frequency dominance。
- **方法**：RiVER 用实例内排序校准 reward，强化 top-ranked solver，同时为其他有效解保留有界反馈，再进行 group-relative RL。
- **效果**：作者报告 Qwen3-8B 与 GLM-Z1-9B-0414 在 ALE rating rank 分别提升 8.9% 和 9.4%，并在 LiveCodeBench、USACO 上平均绝对提升 2.4% 和 3.5%；结果尚待独立复现。[arXiv](https://arxiv.org/abs/2606.27369)

### 2. ⭐ PEEU：用自主探索和 hindsight data 提升小型 GUI Agent
- **背景**：小型开源 MLLM 在隐私和成本上有优势，但跨网站规划弱。
- **问题**：掌握原子 GUI 操作并不自动产生高层任务分解能力，训练数据也常缺少严格对齐的规划轨迹。
- **方法**：PEEU 让 Agent 自主探索环境获取经验，再用 hindsight experience 合成高层规划训练数据；TDHAF 按低、中、高三层分析组合泛化。
- **效果**：论文报告其 7B 模型达到 30.6% accuracy，并超过 Qwen2.5-VL-32B；这是作者在指定 benchmark 上的结果，尚无独立评测。[arXiv](https://arxiv.org/abs/2606.27330)

### 3. ⭐ Sequence probability 不能直接充当正确性判据
- **背景**：beam search、self-consistency 和 verifier-free improvement 往往假设高概率序列更可信。
- **问题**：样本间的概率—正确性相关性，未必能迁移到同一 prompt 的多个回答或不同 decoding 参数。
- **方法**：研究跨模型、benchmark、decoding method 与超参数，在四个层级比较 sequence probability 和 correctness。
- **效果**：作者发现固定数据集的不同问答间常有相关性，但提高序列概率并不能稳定提高准确率，对同一 prompt 的回答也不是可靠置信度；结论提醒评测必须使用外部 verifier 或校准信号。[arXiv](https://arxiv.org/abs/2606.27359)

### 4. ⭐ 仓库视觉结构成为 coding agent 的额外上下文
- **背景**：repo-level Agent 通常只把代码当文本搜索空间，容易丢失目录、模块和依赖的整体结构。
- **问题**：全量塞入 context 成本过高，局部检索又可能漏掉跨模块关系。
- **方法**：论文探索把 repository 的结构化视觉表示与文本内容共同提供给 Agent，以辅助 navigation 和定位。
- **效果**：作者报告在 repo navigation 上改善成本—效果平衡；公开页面不足以证明对所有语言和大型 monorepo 泛化，暂无独立评测。[arXiv](https://arxiv.org/abs/2606.14061)

### 5. 📌 Gemma 4 12B 强化单机可部署路线
- **背景**：企业需要在 frontier API 之外保留本地、可控和较低成本的模型层。
- **问题**：中小模型必须在 memory footprint、吞吐与多任务能力之间平衡。
- **方法**：Google 以 12B 参数开放权重模型延续 Gemma 路线，但完整训练数据构成、推理成本和跨硬件实测仍需结合 model card 判断。
- **效果**：官方公布了产品定位；当前缺少与同级模型统一条件下的独立横评，因此不把厂商指标外推为生产优势。[Google](https://blog.google/innovation-and-ai/technology/developers-tools/introducing-gemma-4-12B/)
