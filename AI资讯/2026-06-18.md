## 🧠 大模型与 LLM 技术解读

- 🔥 **OpenAI Deployment Simulation**
  - 背景：随着 GPT-5 系列进入更强的 reasoning 与 agent 阶段，传统 static eval、synthetic prompt 和红队测试越来越难准确预测真实上线后的失配行为。
  - 问题：上线前如何更接近真实流量地估计 undesired behavior 频率，同时降低模型“知道自己在被测试”带来的评测偏差。
  - 方法：OpenAI 在隐私保护前提下重放既有真实对话前缀，用待发布模型重新生成回复；对 agent 场景则进一步模拟工具调用环境，并在约 130 万条去标识化对话上做风险频率预测。
  - 效果：官方称该方法在 GPT-5 系列 Thinking 部署中把风险率估计做得更准，整体中位误差约 1.5x，并且 simulated traffic 被模型识别为“像评测”的比例接近真实流量；这是目前较少见的“上线前 deployment-like safety forecast”公开方案。来源：[OpenAI Deployment Simulation](https://openai.com/index/deployment-simulation/)

- 🔥 **LifeSciBench**
  - 背景：AI scientist、科学 agent 与 life sciences 专用模型正在快速出现，但多数 benchmark 仍偏向事实问答或狭窄任务，难覆盖真实研发流程。
  - 问题：如何评估模型是否真能处理研究人员在药物发现、生物实验设计和证据审阅中的多步骤、含不确定性任务。
  - 方法：OpenAI 发布由 173 位具 PhD 背景且有 biotech / pharma 经验的科学家参与构建的 benchmark，包含 750 个任务、7 类工作流、7 个生物领域、1062 个附件工件和 19020 条 rubric criteria。
  - 效果：它更像“研究级 workflow eval”而不是传统 QA 榜单，能迫使模型处理 artifact、证据冲突与 operational judgment；但目前它是 benchmark 发布，不是某个模型已经显著领先的独立结论，暂无统一第三方排行榜。来源：[Introducing LifeSciBench](https://openai.com/index/introducing-life-sci-bench/)

- 🔥 **Claude Fable 5 / Mythos 5**
  - 背景：frontier labs 正在把长时程 coding、research 和 cyber agent 作为核心竞争方向，模型不只比 benchmark，也比能否持续完成多小时任务。
  - 问题：如何在保留高强度 autonomy 与 cybersecurity capability 的同时，区分“通用可用版本”和“高风险受控版本”的访问边界。
  - 方法：Anthropic 发布 Fable 5 作为带安全约束的通用版本，同时通过 Project Glasswing 向小范围 cyber defenders 提供同底模、但部分 safeguard 放宽的 Mythos 5；官方强调更长 autonomy、vision、memory 与 life sciences research 能力。
  - 效果：Anthropic 声称 Mythos 5 具备极强网络安全能力，Fable 5 在多项长任务场景中优于过往 Claude；但 6 月 12 日美国出口管制指令要求暂停对外国人的访问，导致公开可验证窗口被打断，因此当前仍应视作“官方声明为主，暂无充分独立评测”。来源：[Claude Fable 5 and Claude Mythos 5](https://www.anthropic.com/news/claude-fable-5-mythos-5) · [Anthropic suspension statement](https://www.anthropic.com/news/fable-mythos-access)

- ⭐ **DiffusionGemma**
  - 背景：本地推理和 interactive agent 工作流越来越重视 latency，传统 autoregressive LLM 的 token-by-token 生成在本地 GPU 上存在明显速度瓶颈。
  - 问题：如何在尽量保留 usable intelligence 的前提下，把文本生成改造成更适合高吞吐、低等待时间的本地交互模型。
  - 方法：Google 发布 Apache 2.0 的实验性 open model DiffusionGemma，采用 text diffusion 路线而非逐 token 自回归；模型为 26B MoE、推理时激活 3.8B 参数，并以 256-token block 并行生成。
  - 效果：Google 宣称在专用 GPU 上可达最高 4x 更快文本生成，H100 上超过 1000 tokens/s，但同时明确其整体输出质量低于标准 Gemma 4，更适合 in-line editing、code infilling 和 speed-critical local workflows；暂无独立大规模第三方评测。来源：[DiffusionGemma: 4x faster text generation](https://blog.google/innovation-and-ai/technology/developers-tools/diffusion-gemma-faster-text-generation/)

- ⭐ **Gemini 3.5 Live Translate**
  - 背景：多模态模型竞争正从“能否听说看写”转向“能否在实时语音交互中低延迟、自然地跨语言执行”。
  - 问题：传统 turn-by-turn speech translation 延迟高、语调损失明显，而且语种对和会议场景扩展能力有限。
  - 方法：Google 用 Gemini 3.5 Live Translate 做连续 speech-to-speech translation，自动识别 70+ 语言，保留说话人的 intonation、pacing 和 pitch，并在生成音频中加入 SynthID watermark。
  - 效果：官方称该模型已向 Gemini Live API / AI Studio、Google Meet 私有预览和 Translate App 推出；Meet 场景支持 2000+ language combinations。当前公开材料未给出独立 latency/WER 对比，因此“可用性提升明确，公开第三方评测仍不足”。来源：[Gemini 3.5 Live Translate](https://blog.google/innovation-and-ai/models-and-research/gemini-models/gemini-live-3-5-translate/)
