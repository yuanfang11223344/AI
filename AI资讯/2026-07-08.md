## 🧠 大模型与 LLM 技术解读

1. 🔥 Claude Sonnet 5：小模型级成本继续承接 agentic 工作流
   - 背景：Claude Code、Codex、Gemini CLI 等 coding/agent 工具把模型从聊天接口推向长任务执行，Sonnet 线承担高频、低成本、可用性更高的生产负载。
   - 问题：企业 agent 最大瓶颈不是单轮 reasoning，而是长任务规划、工具调用、浏览器/终端执行、成本和可预测限额。
   - 方法：Anthropic 将 Sonnet 5 描述为更 agentic 的 Sonnet，可制定计划、调用浏览器和终端，并在 Claude Code 与 Claude Platform 中作为默认/可选模型提供；定价低于 Opus/Fable 级模型。
   - 效果：官方称其在多个 benchmark 上超过 Sonnet 4.6，并接近此前更昂贵模型的 autonomous task 能力；当前主要依据是 Anthropic 官方评测，暂无独立大规模复测。来源：[Anthropic](https://www.anthropic.com/news/claude-sonnet-5)
2. 🔥 GPT-5.6 Sol + Cerebras：前沿模型推理开始和专用硬件联合发布
   - 背景：GPT-5.6 Sol 是 OpenAI 新一代模型预览的一部分，推理吞吐、延迟和可用容量成为企业部署的关键差异。
   - 问题：前沿模型在 coding、agent 和复杂分析中能力提升后，瓶颈转向 token/s、并发容量、成本和受限客户访问。
   - 方法：OpenAI 称将在 7 月把 GPT-5.6 Sol 运行在 Cerebras 上，面向 select customers 提供最高 750 tokens/s 的服务，并和 Terra/Luna 等模型线分层推进。
   - 效果：公开材料给出了速度目标和受限访问计划，但没有公开完整独立 benchmark；需要等 API 可用性、价格、上下文长度和第三方 eval 补齐。来源：[OpenAI](https://openai.com/index/previewing-gpt-5-6-sol/)
3. ⭐ Muse Image / Muse Video：社交上下文驱动的多模态生成模型
   - 背景：Meta 把 Muse 系列放在 Meta AI、Instagram、WhatsApp 入口，目标不是只做模型 API，而是服务社交平台内的创作与广告场景。
   - 问题：图像/视频生成的瓶颈从画质扩展到复杂 prompt 理解、多人/多参考图组合、编辑可控性、社交内容授权与滥用边界。
   - 方法：Meta 称 Muse Image 使用 advanced reasoning 理解复杂 prompts，支持 presets、@ mention 添加照片、手绘编辑；Muse Video 预览强调 prompt adherence、视觉与时间一致性和原生音频。
   - 效果：官方与媒体报道都强调产品可用性，但独立模型卡、训练数据说明和第三方 benchmark 尚不足；隐私和授权机制需要重点观察。来源：[Meta Newsroom](https://about.fb.com/news/2026/07/introducing-muse-image-meta-ai/)、[Meta AI Blog](https://ai.meta.com/blog/introducing-muse-image-muse-video-msl/)
4. ⭐ Gemini Deep Think / Aletheia：把 verifier 放入科学推理 agent 回路
   - 背景：Google DeepMind 正把 Gemini Deep Think 从竞赛数学扩展到科研问题求解，强调 human-AI collaboration。
   - 问题：科研推理任务很难只靠一次性生成答案，模型需要发现候选解中的漏洞，并在失败时回退重试。
   - 方法：DeepMind 描述了内部代号 Aletheia 的数学研究 agent：generator 生成候选解，natural language verifier 识别错误，reviser 修复小问题，严重错误则回到 generator。
   - 效果：官方称 2026 版 Gemini Deep Think 在 IMO-ProofBench Advanced 随 inference-time compute 扩展到约 90% 分数；这些仍以官方论文/博客为依据，需要关注复现和真实科研贡献。来源：[Google DeepMind](https://deepmind.google/blog/accelerating-mathematical-and-scientific-discovery-with-gemini-deep-think/)
5. ⭐ Long-context KV cache compression：DepthWeave-KV、FreqDepthKV、KVpop 同日密集出现
   - 背景：长上下文和 agent 会把 KV cache 变成推理显存和带宽瓶颈，尤其在多轮工具调用、RAG 和代码库理解中明显。
   - 问题：如何减少 KV cache 占用，同时尽量不破坏跨层信息、长距离依赖和生成质量。
   - 方法：`DepthWeave-KV` 使用 token-adaptive cross-layer residual factorization；`FreqDepthKV` 使用 frequency-guided depth sharing；`KVpop` 采用 predictive online pruning。
   - 效果：这些是 2026-07-08/07 附近 arXiv 新条目，标题和摘要显示都指向长上下文推理效率；具体压缩率、困惑度和下游任务退化需阅读全文验证。来源：[arXiv:2607.06523](https://arxiv.org/abs/2607.06523)、[arXiv:2607.06515](https://arxiv.org/abs/2607.06515)、[arXiv:2607.05061](https://arxiv.org/abs/2607.05061)
6. 📌 Danus / DynaKRAG / RSF-GLLM：RAG 与 reasoning agent 正在从单次检索转向结构化控制
   - 背景：多跳问答、数学推理和企业知识问答要求模型不仅“拿到文档”，还要控制证据、记忆事实图和生成过程。
   - 问题：RAG 容易出现证据漂移、检索冗余、生成与结构化知识脱节，以及 multi-hop reasoning 不可解释。
   - 方法：`Danus` 强调 fact-graph memory 编排数学 reasoning agents；`DynaKRAG` 提出 learnable evidence control；`RSF-GLLM` 用 recurrent soft-flow 和 decoupled LLM generation 缩小 KG QA 语义鸿沟。
   - 效果：公开 arXiv 条目显示 RSF-GLLM 已被 ICML 2026 接收；其余效果需以论文实验为准，不能仅凭标题外推。来源：[arXiv:2607.06447](https://arxiv.org/abs/2607.06447)、[arXiv:2607.06507](https://arxiv.org/abs/2607.06507)、[arXiv:2607.06527](https://arxiv.org/abs/2607.06527)
