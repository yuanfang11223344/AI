## 🧠 大模型与 LLM 技术解读

### 1. Claude Fable 5 / Mythos 5
- 背景：Anthropic 在 6 月 9 日发布 `Claude Fable 5` 与 `Claude Mythos 5`，但 6 月 12 日即因出口管制暂停访问，说明 frontier model 发布已经嵌入国家级治理框架。  
- 问题：前沿模型一方面追求更强通用推理与 agent/coding 能力，另一方面需要满足可控部署和合规约束。  
- 方法：Anthropic 将 `Fable 5` 定位为 “Mythos-class model made safe for general use”，即在高能力基线之上通过安全化处理面向更广泛用户开放。  
- 效果：目前公开可确认的是发布与暂停两件事本身；但对外尚缺少完整技术报告、系统卡和独立 benchmark 细节，暂无独立评测。  
- 来源：[Anthropic 发布页](https://www.anthropic.com/news/claude-fable-5-mythos-5)、[暂停访问说明](https://www.anthropic.com/news/fable-mythos-access)

### 2. GLM-5.2 长任务能力升级
- 背景：长上下文和 long-horizon task 正成为 agent、research assistant、coding harness 的核心能力要求，仅有聊天 benchmark 已不足以区分模型。  
- 问题：模型往往能“看见”长上下文，却无法稳定完成多步骤任务、状态跟踪和长链条执行。  
- 方法：Z.ai 在 Hugging Face 公布 `GLM-5.2`，强调 `1M-token context` 与 long-horizon task 提升，把上下文窗口和任务执行一起作为能力目标。  
- 效果：官方称相较 `GLM-5.1` 有明显跃升，但目前主要依据发布材料，缺少第三方系统性复现与独立评测。  
- 来源：[Hugging Face / Z.ai](https://huggingface.co/blog/zai-org/glm-52-blog)

### 3. GPT-5.5 面向执行型知识工作
- 背景：LLM 竞争已从单题推理转向“能否持续完成复杂工作流”，尤其是 coding、research、data analysis 和 document-heavy 场景。  
- 问题：高端模型常在长任务中出现中途偏航、工具协作不稳、延迟过高或 token 成本过高的问题。  
- 方法：OpenAI 在 `GPT-5.5` 中强调更好的 task persistence，并给出多个 execution-oriented benchmark，如 `GDPval`、`OSWorld-Verified`、`Tau2-bench Telecom`。  
- 效果：OpenAI 公布 `GDPval 84.9%`、`OSWorld-Verified 78.7%`、`Tau2-bench Telecom 98.0%`，并称其在 scientific workflow 上较 `GPT-5.4` 提升明显；但这些结论目前仍主要来自官方材料。  
- 来源：[OpenAI GPT-5.5](https://openai.com/index/introducing-gpt-5-5/)

### 4. ARD: 让 Agent 在运行时搜索资源
- 背景：MCP 与 tool ecosystem 快速扩张后，agent 最大瓶颈之一变成“知道有什么工具、去哪找、怎么发现”。  
- 问题：预装式工具清单难以覆盖开放生态，agent 缺乏统一 discovery layer 时，跨组织和跨 registry 的可扩展性很差。  
- 方法：Hugging Face 介绍 `Agentic Resource Discovery (ARD)` 草案，作为位于 registry 前方的 discovery layer，用于 catalog、index、search tools/capabilities。  
- 效果：ARD 的价值不在单个模型分数，而在生态兼容性；它由 Microsoft、Google、GoDaddy、Hugging Face 等共同参与，但目前仍是 draft spec，暂无大规模生产验证。  
- 来源：[Hugging Face ARD](https://huggingface.co/blog/agentic-resource-discovery-launch)

### 5. ChatGPT Dreaming 记忆系统
- 背景：用户越来越希望 agent 能记住长期偏好、项目上下文与历史决策，而不是每次从零开始。  
- 问题：没有统一 memory foundation 时，个性化与持续协作很难稳定，且跨任务迁移容易丢上下文。  
- 方法：OpenAI 将 `Dreaming` 描述为共享记忆底座，让记忆系统更持续地服务多轮协作与偏好保持。  
- 效果：官方表述是“most capable memory system yet”；但对外公开的技术细节有限，尚无独立对照评测来量化其对任务完成率的提升。  
- 来源：[OpenAI Dreaming](https://openai.com/index/chatgpt-memory-dreaming/)

## 📄 论文速递

- [arXiv:2606.19347](https://arxiv.org/abs/2606.19347) `How LLMs Fail and Generalize in RTL Coding for Hardware Design?`：把 LLM 放进 RTL coding 这一更接近 ASIC 前端设计的场景，和你的工作画像直接相关，值得关注模型在哪些硬件设计模式上稳定失效。
- [arXiv:2606.19348](https://arxiv.org/abs/2606.19348) `DeepSeek-V4: Towards Highly Efficient Million-Token Context Intelligence`：继续围绕 million-token context 做效率与能力叙事，适合和 GLM-5.2 一起观察长上下文路线是否收敛。
- [arXiv:2606.19544](https://arxiv.org/abs/2606.19544) `Reliability without Validity`：系统评估 `LLM-as-a-Judge` 在 agreement、consistency、bias 上的问题，适合所有依赖自动评测和 synthetic evaluation 的团队。
- [arXiv:2606.19667](https://arxiv.org/abs/2606.19667) `CacheWeaver`：针对 grounded RAG inference 做 cache-aware evidence ordering，关注点是把检索与推理时延压到更实用的系统区间。
- [arXiv:2606.20205](https://arxiv.org/abs/2606.20205) `Apparent Psychological Profiles of Large Language Models are Largely a Measurement Artifact`：对“给模型做人格测评”这类流行方法提出质疑，提醒团队别把测量伪影当成模型内在属性。
- [arXiv:2606.20014](https://arxiv.org/abs/2606.20014) `Hierarchical Control in Multi-Agent Games: LLM-based Planning and RL Execution`：把 LLM 规划与 RL 执行结合，适合观察 multi-agent 体系中 planner/executor 分层是否能稳定提升可控性。
