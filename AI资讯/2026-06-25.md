## 🧠 大模型与 LLM 技术解读

### 1. 🔥 Gemini 3.5 Flash 原生 computer use：agent 能力从外接工具转向模型内建

- **背景**：Google 5 月发布 Gemini 3.5 Flash，定位为面向 agent 和 coding 的高速 frontier model；6 月 24 日又把原先独立的 computer use capability 合入主 Flash 模型。
- **问题**：浏览器/桌面/移动端自动化长期受制于视觉理解、动作规划、工具调用安全和 prompt injection 风险。外接 computer-use 模型会增加编排复杂度，也更难与主模型推理上下文统一。
- **方法**：Gemini 3.5 Flash 现在可以直接“see, reason and take action” across browser/mobile/desktop；企业侧提供敏感动作显式确认和检测 indirect prompt injection 后自动停止的 safeguard。Google 还给出 reference implementation 和 Browserbase demo。
- **效果**：Google 声称它是 Gemini 当前最好的 agentic computer use 表现，并适合连续软件测试、知识工作自动化等长周期任务；公开材料主要为官方 benchmark 与客户案例，暂无独立评测。来源：[Google Blog](https://blog.google/innovation-and-ai/models-and-research/gemini-models/introducing-computer-use-gemini-3-5-flash/)

### 2. 🔥 DiffusionGemma：用 diffusion 解码挑战 autoregressive LLM 的本地延迟瓶颈

- **背景**：主流 LLM 逐 token autoregressive 生成在云端高并发场景可通过 batching 提高利用率，但本地单用户推理常受 memory bandwidth 与 sequential decode 限制。
- **问题**：本地 interactive workflow（inline editing、快速代码补全、低并发私有部署）需要低延迟和高 tokens/s，传统 AR 解码在单卡上容易让 GPU 空等。
- **方法**：Google 发布 Apache 2.0 的 26B MoE DiffusionGemma，推理时激活 3.8B 参数，通过 diffusion head 一次并行生成 256-token block，再迭代修正整段文本；双向 attention 让 token 可互相约束。
- **效果**：Google 称在专用 GPU 上可达最高 4x text generation speed，H100 单卡 1000+ tokens/s、RTX 5090 700+ tokens/s；但官方也明确输出质量低于标准 Gemma 4，推荐将其用于 speed-critical local workflows。来源：[Google Blog](https://blog.google/innovation-and-ai/technology/developers-tools/diffusion-gemma-faster-text-generation/)

### 3. ⭐ Gemma 4 12B：encoder-free multimodal 尝试把音频/视觉直接并入 LLM backbone

- **背景**：开源/开放权重模型正在往可本地运行的 multimodal agent 方向推进，用户需要在 laptop/edge 上运行视觉、音频、文本统一模型。
- **问题**：传统 multimodal architecture 常使用独立视觉/音频 encoder，再把 embedding 交给 LLM；这会增加延迟、显存占用和 pipeline 复杂度。
- **方法**：Gemma 4 12B 采用统一、encoder-free 设计：视觉以轻量 embedding module 替代视觉 encoder，音频则将 raw audio 投影到文本 token 同一空间；模型支持 MTP drafters 降低 latency，并可在 16GB VRAM/统一内存设备上运行。
- **效果**：Google 称 Gemma 4 12B 在标准 benchmark 上接近 26B MoE，Gemma 4 系列累计下载超过 1.5 亿次；具体跨任务独立评测仍需社区复现。来源：[Google Blog](https://blog.google/innovation-and-ai/technology/developers-tools/introducing-gemma-4-12b/)

### 4. 🔥 GPT-5.5-Cyber / Codex Security：LLM 从辅助审计走向可验证修补 pipeline

- **背景**：前沿模型已经能在大型代码库中发现漏洞，产业瓶颈从“找不到漏洞”转向“验证、分级、披露、修补和回归测试跟不上”。
- **问题**：如果 AI 只提升攻击者发现漏洞的速度，而没有同步提高维护者修补速度，开源和基础设施安全窗口会被压缩。
- **方法**：GPT-5.5-Cyber 面向可信防御者开放；Codex Security 插件可做 deep scan、attack path、threat model、validation evidence 和 codebase-specific patch；Patch the Planet 则把安全工程师、维护者、企业伙伴纳入修补闭环。
- **效果**：The Hacker News 报道称初始参与项目包括 cURL、NATS Server、pyca/cryptography、Sigstore、aiohttp、Go、freenginx、Python 等；公开材料强调治理和 human oversight，暂无第三方统一 benchmark。来源：[The Hacker News](https://thehackernews.com/2026/06/openai-expands-daybreak-with-gpt-55.html)

### 5. ⭐ Claude Tag：从个人 Claude Code 迁移到多人上下文中的 agent runtime

- **背景**：Claude Code/Cowork 等个人或单任务 agent 工具已证明对工程工作有价值，但团队协作需要共享上下文、可见任务状态和权限边界。
- **问题**：单人 chat/IDE agent 很难天然承载团队记忆、跨成员接续、公共可审计输出，也容易造成任务重复或上下文碎片。
- **方法**：Claude Tag 把 Claude 放进 Slack channel：用户通过 `@Claude` 发起任务，Claude 分阶段规划、调用可用工具并在线程里汇报，channel 中的其他人可以看到进展并接续。
- **效果**：Anthropic 披露其内部产品团队 65% 代码由内部版 Claude Tag 生成；外部版本仍为 Claude Enterprise/Team beta，暂无独立生产力评测。来源：[Anthropic](https://www.anthropic.com/news/introducing-claude-tag)
