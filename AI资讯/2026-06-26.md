## 🧠 大模型与 LLM 技术解读

### 1. 🔥 Gemini 3.5 Flash 原生 computer use：agent 能力从外挂工具变成主模型内建能力

- **背景**：2026 年的 agent 竞争已经从 function calling 走向真实 GUI / browser / desktop execution，Google 在 6 月 24 日把 computer use 并入 Gemini 3.5 Flash。
- **问题**：单独的 computer-use 模型会增加编排复杂度，也让视觉理解、动作规划和主模型上下文分裂；同时 prompt injection 与高风险点击操作难以统一治理。
- **方法**：Google 把 computer use 作为 Gemini 3.5 Flash 的 built-in tool，支持在浏览器、移动端和桌面环境中感知界面并执行动作，同时加入敏感操作确认与 indirect prompt injection 检测。
- **效果**：Google 声称这是其当前最佳的 agentic computer-use 表现，并给出 Browserbase demo 与参考实现；当前公开证据主要来自官方材料，暂无系统性的独立评测。来源：[Google Blog](https://blog.google/innovation-and-ai/models-and-research/gemini-models/introducing-computer-use-gemini-3-5-flash/)

### 2. ⭐ DiffusionGemma：用 diffusion block decoding 改写本地 LLM 推理速度边界

- **背景**：autoregressive LLM 在单用户、本地、低并发环境里常被 sequential decode 和 memory bandwidth 卡住，边缘设备很难既保留模型质量又获得足够 tokens/s。
- **问题**：本地 coding、编辑和私有部署需要低延迟，但逐 token 解码让 GPU/统一内存系统利用率不高，速度成本都不理想。
- **方法**：Google 发布 26B MoE 的 DiffusionGemma，推理时激活约 3.8B 参数，用 diffusion 方式并行生成 256-token block，再进行迭代修正，而不是严格逐 token 自回归。
- **效果**：Google 宣称文本生成速度可最高提升到 4x，并给出 H100 与 RTX 5090 上的高吞吐示例；但官方也明确表示输出质量仍低于标准 Gemma 4，适合 speed-critical local workflows，暂无独立大规模复现。来源：[Google Blog](https://blog.google/innovation-and-ai/technology/developers-tools/diffusion-gemma-faster-text-generation/)

### 3. ⭐ Gemma 4 12B：encoder-free multimodal 向 laptop 级 agentic multimodal intelligence 推进

- **背景**：开源/开放权重模型正在从纯文本助手转向可本地部署的 multimodal agent，用户希望在单机侧跑视觉、音频和文本统一模型。
- **问题**：传统 multimodal 方案依赖独立视觉/音频 encoder，系统延迟、显存占用与工程复杂度都偏高，不利于 edge 和 laptop 场景。
- **方法**：Gemma 4 12B 采用 unified、encoder-free 设计，视觉用轻量 embedding module，音频直接投影到统一 token 空间，并加入 native audio inputs。
- **效果**：Google 将其定位为介于 E4B 与 26B MoE 之间的中型 multimodal 模型，并称 Gemma 4 全系下载已超过 1.5 亿次；公开材料强调设备侧效率与能力平衡，但暂无完整第三方基准汇总。来源：[Google Blog](https://blog.google/innovation-and-ai/technology/developers-tools/introducing-gemma-4-12b/)

### 4. 🔥 Thinking to Recall：Google Research 指出 reasoning token 不只做复杂推理，也会提升事实召回

- **背景**：链式思维通常被视为解决复杂推理问题的方法，但对单跳事实问答是否有帮助一直不清楚。
- **问题**：如果一个事实本来就在模型参数里，为什么额外 reasoning token 会提高命中率？这直接关系到 inference budget、test-time scaling 和评测解释。
- **方法**：Google Research 对 reasoning on/off 进行受控对比，并提出两种机制解释：一是 reasoning token 提供额外 latent computation buffer，二是中间相关事实会对最终答案形成 factual priming。
- **效果**：研究显示 reasoning 打开后，模型能召回原本几乎不可达的简单事实；同时如果中间轨迹出现 hallucinated fact，最终正确率会显著下降。该结论来自研究博客与论文摘要，仍需更多外部复现实验。来源：[Google Research](https://research.google/blog/thinking-to-recall-how-reasoning-unlocks-parametric-knowledge-in-llms/)

### 5. ⭐ GPT-5.5：OpenAI 继续把长上下文、tool use 和 knowledge work 一起产品化

- **背景**：frontier model 的竞争重点正从单轮聊天转向长任务执行、跨工具协作与专业知识工作自动化。
- **问题**：企业关心的不只是 benchmark 分数，而是模型能否在长上下文、真实文件和多步骤流程中持续工作，并把 token 成本控制在可接受范围。
- **方法**：OpenAI 在 ChatGPT、Codex 与 API 同步推进 GPT-5.5，强调更强的 agentic coding、tool coordination、knowledge work 与 scientific research workflows。
- **效果**：OpenAI 公布了 Terminal-Bench 2.0、OSWorld-Verified、GDPval 等多项指标提升，并给出 1M context window 与 API 定价；这些数据目前仍以官方评测为主，暂无统一独立评测。来源：[OpenAI](https://openai.com/index/introducing-gpt-5-5/)

## 📄 论文速递

1. ⭐ **NatureBench: Can Coding Agents Match the Published SOTA of Nature-Family Papers?**：把 90 个 Nature-family 科学任务做成 benchmark，结论是当前 coding agents 更像“方法迁移器”而非真正 scientific discovery engine。arXiv：[2606.24530](https://arxiv.org/abs/2606.24530)

2. ⭐ **OpenThoughts-Agent: Data Recipes for Agentic Models**：公开 agentic model 的数据构建 pipeline，并报告在 7 个 agent benchmark 上较现有开放数据方案有平均提升。arXiv：[2606.24855](https://arxiv.org/abs/2606.24855)

3. ⭐ **Tmax: A simple recipe for terminal agents**：提出面向 terminal agents 的开放 RL recipe，9B 模型在 Terminal-Bench 2.0 上达到 27%，主打“简单但够强”的开源基线。arXiv：[2606.23321](https://arxiv.org/abs/2606.23321)

4. ⭐ **PlanBench-XL: Evaluating Long-Horizon Planning of LLM Tool-Use Agents in Large-Scale Tool Ecosystems**：用 327 个 retail tasks 和 1,665 个 tools 测试 agent 在大工具生态中的检索、规划与重规划能力。arXiv：[2606.22388](https://arxiv.org/abs/2606.22388)

5. ⭐ **How Inference Compute Shapes Frontier LLM Evaluation**：指出 frontier benchmark 分数对 token budget、context compaction 和 repeated submission 高度敏感，提醒不要把单一 budget 下的低分直接当成能力上限。arXiv：[2606.17930](https://arxiv.org/abs/2606.17930)

6. 📌 **LLM Agents Can See Code Repositories**：研究把 repo 结构图作为视觉输入辅助 coding agent，结果显示“文本 + 视觉结构”能减少 token 消耗并改善大型仓库定位。arXiv：[2606.14061](https://arxiv.org/abs/2606.14061)
